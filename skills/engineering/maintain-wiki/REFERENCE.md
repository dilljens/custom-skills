# Maintain Wiki — Reference

## Manifest format (`_hashes.json`)

```jsonc
{
  "config": {
    "include": ["src/**/*.ts", "src/**/*.tsx"],
    "exclude": ["**/*.test.*", "**/__tests__/**", "**/*.d.ts"],
    "language": "typescript",
    "workspace_root": "src/",
    "existing_docs": [
      { "path": "README.md", "covers": "project overview, setup" },
      { "path": "CONTEXT.md", "covers": "domain language, architecture decisions" }
    ]
  },
  "domains": {
    "features/task-system": {
      "label": "Task System",
      "description": "Task generation, assignment, completion, minigames"
    },
    "features/meetings": {
      "label": "Meetings",
      "description": "Emergency meetings, voting, ejection"
    }
  },
  "files": {
    "src/lib/tasks.ts": { "hash": "abc123def456", "doc": "features/task-system" },
    "src/hooks/useTasks.ts": { "hash": "789ghi012jkl", "doc": "features/task-system" },
    "src/lib/meetings.ts": { "hash": "345mno678pqr", "doc": "features/meetings" },
    "src/components/NewFeature.tsx": { "hash": null, "doc": null }
  }
}
```

- `doc: null` means the file is tracked but not assigned to a domain yet (undocumented).
- `hash: null` means the file exists on disk but isn't in the git tree yet (untracked/new).
- Hash is the full 40-char git blob hash from `git ls-tree -r HEAD`.

## Module doc template

Every domain doc at `docs/wiki/<domain>.md` follows this structure:

```markdown
# <Domain Label>

**Source**: `<file1>`, `<file2>`, ...
**Last updated**: `YYYY-MM-DD`

Files tracked for this domain (...N files)

## What it does
2-4 sentences. Purpose, role in the application, when it's used.

## Architecture
How the pieces connect. State ownership, dependency direction, key interfaces.
Include a simple ASCII data-flow diagram if the flow is non-trivial.

## Key functions / components
| Name | Location | Purpose |
|------|----------|---------|
| `generateTasks()` | lib/tasks.ts:42 | Assigns randomized tasks to each player at game start |
| `useTasks()` | hooks/useTasks.ts:18 | React hook with Supabase realtime, exposes task state |

## Data flow
Numbered step-by-step: what triggers this module, what it does, what it emits.

## Edge cases & gotchas
- Unexpected states, race conditions, refresh-safety concerns
- Interactions with other modules (sabotage affecting tasks, etc.)

## Dependencies
| Depends on | For |
|------------|-----|
| `lib/supabase.ts` | Realtime task updates |
| `lib/stations.ts` | Station lookup by QR |
```

### Tech stack adaptations

| Language | Extra section | Content |
|----------|--------------|---------|
| React/Next.js | **Components** | Table of components with props summary |
| React/Next.js | **Hooks** | Table of hooks with return values |
| Go | **Commands / Handlers** | CLI commands or HTTP handlers |
| Python API | **Endpoints** | Route/method/purpose table |
| Rust | **Types / Traits** | Key structs, enums, trait impls |

Detect the stack from config files and auto-add the right sections. The base sections (What it does, Architecture, Data flow, Edge cases, Dependencies) always apply.

## `_deps.md` template

Generated from import analysis. Shows blast radius for any change:

```markdown
# Dependency Map

## Module dependency graph
| Module | Depends on | Depended on by |
|--------|-----------|----------------|
| `infrastructure/supabase` | — | *everything* |
| `infrastructure/realtime` | `infrastructure/supabase` | `features/*`, `ui/*` |

| `features/meetings` | `infrastructure/supabase` | `ui/game-board` |
| `ui/game-board` | `features/*`, `hooks/*` | — |

## High-risk modules
Modules depended on by 3+ others — changes here have high blast radius:
- `infrastructure/supabase` (depended on by 8 modules)

Modules with no dependents — safe to refactor freely:
- `features/sabotage`
```

Detection: scan every source file for import/require/include statements, build a directed graph, classify modules by dependency count. Regenerate on every `update wiki`.

## `_patterns.md` template

Code archeology — how things are done in this project. Editorial (not hash-tracked). Created during `make wiki`, manually refreshed with `refresh patterns`.

```markdown
# Code Patterns

Detected conventions. Before writing new code, match these patterns.

## Error handling
**Pattern**: try/catch in lib → return `{ data, error }` tuple → hook surfaces via state
**Canonical example**: `src/lib/games.ts:createGame()` — returns `{ data: Game, error: null }` on success, `{ data: null, error: msg }` on failure
**Stack rule**: never throw across module boundary

## Async flow
**Pattern**: async function in lib → hook calls it in useEffect → stores result in useState
**Canonical example**: `src/hooks/useGame.ts` — useEffect calls `loadGame()`, sets `game` state
**Rule**: hooks own async lifecycle, lib functions are stateless

## Module structure
**Pattern**: barrel export via `index.ts` → import from `@/lib`, `@/hooks`
**Canonical example**: `src/lib/index.ts` re-exports all lib functions
**Rule**: never import from internal paths; use barrel

## State management
**Pattern**: React useState + Supabase realtime subscription for cross-client sync
**Canonical example**: `src/hooks/usePlayers.ts` — local state mirrors DB, realtime pushes updates
**Rule**: one hook per realtime channel, cleanup subscriptions on unmount

## Naming conventions
| Kind | Convention | Example |
|------|-----------|---------|
| Files | kebab-case | `use-game.ts`, `game-board.tsx` |
| Functions | camelCase | `createGame()`, `usePlayers()` |
| Components | PascalCase | `GameBoard`, `PlayerList` |
| Types/Interfaces | PascalCase | `Game`, `PlayerState` |
| DB tables | snake_case | `player_tasks`, `kill_records` |
| DB columns | snake_case | `game_id`, `is_completed` |

## Component pattern
**Pattern**: `'use client'` → import hooks → destructure → render JSX
**Canonical example**: `src/components/game/PlayerList.tsx`
**Rule**: presentational only — no fetch, no business logic, no DB access

## Test pattern
**Pattern**: vitest + jsdom, co-located in `src/__tests__/`, one file per source module
**Rule**: test behavior not implementation, use testing-library matchers
```

### Detection heuristics (generalized)

Patterns are detected by scanning a representative sample of source files (3–5 per layer: lib, hooks, components, handlers, etc.) and looking for:

| Pattern category | Detection method |
|-----------------|------------------|
| Error handling | Count throw vs return patterns; detect `Result`, `Either`, `{ data, error }` tuples |
| Async flow | Detect async/await, Promise chains, callbacks, event emitters, CSP channels |
| Module structure | Read barrel files (index.ts/__init__.py/mod.rs); detect import style (named, default, wildcard) |
| State management | Detect React hooks, stores (Redux, Zustand, Pinia), signals, actors |
| Naming conventions | Sample 20+ file/fn/type names from each layer; detect dominant style |
| Component/data pattern | Detect `'use client'`, decorators, annotations, HOCs |
| Test pattern | Read a test file; detect framework + assertion style |

**After detecting patterns, present them to the user for approval** — the user may correct, rename, or reject any pattern. The patterns doc should capture the project's *actual* conventions, not what a linter says.

### Stack-specific patterns

| Stack | Always check |
|-------|-------------|
| React/Next.js | Components (`'use client'`, server vs client), hooks vs server actions |
| Vue/Nuxt | Composables, `<script setup>`, Pinia stores |
| Svelte/SvelteKit | Stores, `$:`, `+page.server.ts` vs `+page.ts` |
| Go | Error wrapping (`%w`), `database/sql` patterns, middleware chaining |
| Python | `async def` vs sync, pydantic models, FastAPI dependency injection |
| Rust | `Result<T, E>`, `?` operator, trait bounds, module visibility |
| CLI tools | Flag parsing (clap, cobra, argparse), subcommand routing |

## `_lifecycle.md` template

State machine documentation. Editorial (not hash-tracked). Created during `make wiki`, manually refreshed with `refresh lifecycle`.

```markdown
# Runtime Lifecycle

## Entry points
| Access | Path | What happens |
|--------|------|-------------|
| Browser | `/` | Lobby page — create/join game |
| Browser | `/game/[id]` | Active game — loads game state |
| Realtime | `games:id` | Game state changes pushed from DB |
| CLI | `npm run simulate` | Headless game simulation |

## Core state machine
```
Lobby → Setup → Playing → (Meeting → Playing)* → GameOver
                           ↓
                      Sabotage → ImposterWin
```

States and transitions:

| State | Trigger | → Next state |
|-------|---------|--------------|
| `lobby` | Host starts game | `setup` |
| `setup` | All tasks assigned | `playing` |
| `playing` | Body reported / emergency meeting | `meeting` |
| `meeting` | Vote complete | `playing` or `game_over` |
| `playing` | All tasks complete | `game_over` |
| `playing` | Sabotage timer expires | `game_over` (imposter win) |
| `playing` | All imposters ejected | `game_over` |

## Entity lifecycles

### Player
```
Join → AssignedRole → Playing
  ├── CompleteAllTasks → WinCheck
  ├── Killed → DeadObserver
  └── CallMeeting → Vote → Eject → DeadObserver | Continue
```

### React component lifecycle
```
Page mount → useGame(gameId) → subscribe to realtime
  ├── game state update → re-render
  ├── page unload → save to sessionStorage → unsubscribe
  └── refresh → restore from sessionStorage → re-subscribe
```

Refresh safety: player data backed up to sessionStorage on unload, recreated on page load.

### Realtime subscriptions
| Channel | Filter | Triggers rerender |
|---------|--------|-------------------|
| `games` | `id` | Game state, status, sabotage |
| `players` | `game_id` | Joins, roles, kills, task completion |
| `player_tasks` | `player_id` | Task completion |
| `meetings` | `game_id` | Meeting lifecycle |
| `votes` | `meeting_id` | Voting progress |
| `sabotages` | `game_id` | Sabotage creation, fix progress |
```

### Detection heuristics (generalized)

| Step | Method |
|------|--------|
| Find entry points | Read config files for main/routes/commands; scan pages/routes/commands directories |
| Detect state machines | Search for state enums (`enum State`, `type Status`), state transitions (switch/match on status), state validation (`isPlaying()`, `canStart()`) |
| Detect entity lifecycles | Find entity types (Player, Game, Order, User) with status fields; trace status changes through code |
| Detect request/event lifecycles | For web: scan middleware chains, app routers; for CLI: trace arg parsing to execution; for APIs: route registration to handler |
| Detect subscriptions/background | Scan for websocket, SSE, polling, realtime subscriptions, cron jobs, workers |

**After detecting, present the state diagram to the user for approval.** The user may fill in gaps the code doesn't explicitly state (e.g., transition conditions that are spread across multiple files).

## Domain detection heuristics

When proposing domains, use these signals (strongest first):

1. **Directory clusters** — `src/features/` or domain-named subdirectories are the strongest signal.
2. **Import communities** — use `rg "import.*from.*\.\." <dir>` to find groups of files that import each other heavily; these are one domain.
3. **Co-location** — files with similar naming in the same directory (e.g., `useGame.ts`, `GameBoard.tsx`, `gameLogic.ts`) are one domain.
4. **Barrel exports** — files re-exported from the same `index.ts` are related.
5. **Config-defined modules** — monorepo packages, workspace members.

Rule of thumb: a domain should cover 2-20 source files. Fewer than 2 is too fine-grained; merge it. More than 20 is too broad; split it.

## File scope defaults

| Language | Default include | Default exclude |
|----------|----------------|-----------------|
| TypeScript/JavaScript | `src/**/*.{ts,tsx,js,jsx}` | `**/*.test.*`, `**/__tests__/**`, `**/*.d.ts`, `**/*.stories.*` |
| Python | `**/*.py` | `**/test_*.py`, `**/tests/**`, `**/conftest.py` |
| Go | `**/*.go` | `**/*_test.go` |
| Rust | `src/**/*.rs` | — |
| Generic | `src/**/*` | `**/test*/**`, `**/__test*/**`, `**/*.test.*`, `**/*.spec.*`, `**/dist/**`, `**/build/**`, `**/.generated/**` |

Always exclude `node_modules/`, `.git/`, `vendor/`, `target/`, `__pycache__/`.

The user can override these in the `config` section of `_hashes.json`.

## Hash diff mechanics

```
git ls-tree -r HEAD -- <include-paths>
```

Output format: `<mode> blob <hash>\t<path>`

```
100644 blob abc123def456...\tsrc/lib/tasks.ts
100644 blob 789ghi012jkl...\tsrc/hooks/useTasks.ts
```

Compare each entry against `_hashes.json.files[path].hash`:

| Current hash | Manifest hash | Manifest doc | Category |
|-------------|---------------|-------------|----------|
| matches | matches | exists | **unchanged** |
| differs | differs | exists | **stale** |
| differs | differs | null | **undocumented** (needs domain assignment) |
| present | N/A (not in manifest) | N/A | **new file** (add to manifest as `null`) |
| N/A (not in git) | present | exists | **orphaned** (file deleted) |

## README.md template

When creating `docs/wiki/README.md`, use:

```markdown
# Codebase Wiki

This wiki mirrors the source tree. Every page is keyed to git blob hashes so
staleness is detectable. When source files change, their hash no longer matches
the manifest and the corresponding wiki page is flagged for update.

## For humans

1. Start at `_index.md` — it covers the full architecture in 5-10 minutes.
2. Drill into domain docs under `features/`, `ui/`, `infrastructure/`, etc.
3. Each doc is 1-3 pages of prose + tables. Read it instead of the source.

## For AI agents

1. Before reading unfamiliar source: check `_index.md` to find the right domain,
   then read the domain doc.
2. If a domain doc feels stale after reading source, mention it with
   `wiki status` or `update wiki`.
3. Manifest: `_hashes.json` maps each source file → domain doc + last-known
   git blob hash. A hash mismatch means the doc is stale.
4. When updating a doc: diff only the changed files, surgically update the
   affected sections. Never rewrite the whole doc from scratch.
5. New files with `doc: null` need a domain assigned. Ask the human.
6. Template for domain docs — REFERENCE.md has the full spec.
7. `_patterns.md` tells you how to write code here. Read it before adding anything new.
8. `_lifecycle.md` tells you how things run. Read it before debugging state.
9. `_tests.md` tells you what's tested and where gaps are. Check before refactoring.
10. `_deps.md` tells you blast radius. Check before changing a module used by others.

## Commands

- `make wiki` — initialize docs/wiki/ (run once per project)
- `update wiki` — detect stale docs via hash diff, update them
- `wiki status` — report staleness without making changes
- `refresh patterns` — re-scan and update _patterns.md
- `refresh lifecycle` — re-scan and update _lifecycle.md
```

## Edge cases

### New project (no docs)
`make wiki` builds everything from scratch. Existing README.md is linked, not duplicated.

### Project with existing docs folder
`make wiki` detects existing `docs/` content, links it from `_index.md`, excludes those modules from the manifest so the wiki fills gaps instead of competing.

### File moved without content change
`git ls-tree` uses blob hashes, so a rename with identical content keeps the same hash. Not detected as stale. The manifest path is wrong though — the diff reports it as orphaned (old path gone) + new file (new path). The agent should update the manifest path without rewriting the doc.

### File deleted
Detected as orphaned. Report to user. If the entire domain is gone, remove its doc. If only one file of many is gone, update the doc's source list and regenerated the affected sections.

### Large refactor touching many files
`update wiki` will flag many stale docs. Present the full list and let the user choose: update all, update specific domains, or `make wiki` fresh (if the domain structure itself changed).

### Monorepo
`make wiki` detects workspace configs (`pnpm-workspace.yaml`, `lerna.json`, `turbo.json`, Cargo workspace, Go workspace). Each workspace member gets its own `docs/wiki/<member>/` with independent manifest. The root gets a meta `_index.md` linking to all members.

### Merge conflicts in _hashes.json
Hash manifests will conflict on merge. Since hashes are content-derived, the resolution is: accept the merged file list, then run `update wiki` to recompute all hashes. The skill should detect this (hashes don't match for all files) and offer to re-seed.
